üîß Backend Data Operations

Understanding how the system processes your data and makes intelligent routing decisions.

üìù Single-Entry Configuration

Enter Once, Use Everywhere: All configuration fields (addresses, API credentials, journey settings) are stored in user-preferences.json on the server. Once saved, these values automatically populate across all modules.

Data Flow:
1. You configure addresses in User Preferences (top section)
2. System saves to persistent JSON file
3. Route Planner auto-loads these addresses (readonly fields)
4. Multi-Modal Router uses saved PTV API credentials
5. Cafe Detector uses saved cafe address
6. All modules stay in sync automatically

üó∫Ô∏è Route Planning Algorithm

Module: route-planner.js

Calculation Process:
1. Geocoding - Converts your addresses to GPS coordinates (OpenStreetMap Nominatim API)
2. Walking Distance - Calculates walking time using Haversine formula (80m/min = 4.8km/h)
3. Cafe Busy-ness - Checks current time against peak periods, adjusts coffee wait time (2-8 min)
4. Backward Calculation - Works backwards from your arrival time:
Arrival 09:00
- Walk to work (5 min) = 08:55 arrive station
- Train (20 min) = 08:35 depart origin
- Walk to station (3 min) = 08:32 leave cafe
- Coffee (3 min) = 08:29 arrive cafe
- Walk to cafe (4 min) = 08:25 leave station
- Walk to station (8 min) = 08:17 LEAVE HOME

Considered Variables: Walking speed (configurable), coffee purchase time (dynamic based on busy-ness), safety buffers (2 min per connection), transit journey duration

üöÜ Multi-Modal Transit Search

Module: multi-modal-router.js

Search Process:
1. Query All Modes - Searches PTV API for: Trains (type 0), Trams (1), Buses (2), V/Line (3)
2. Fetch Departures - Gets next 10 departures for each enabled mode
3. Filter by Time - Only keeps departures within ¬±10 min of required departure time
4. Sort by Proximity - Orders by closest match to ideal departure time
5. Return Best 2 - Shows the 2 best options across all transit modes

Authentication: Uses your saved PTV API credentials with HMAC-SHA1 signature
Cache: Results cached for 30 seconds to reduce API calls
Coffee Feasibility: Each option marked if you have time for coffee before departure

‚òï Cafe Busy-ness Detection

Module: cafe-busy-detector.js

Detection Methods:
1. Google Places API (if configured) - Real-time "popular times" data from Google
2. Time-Based Fallback (always available) - Estimates based on peak periods

Peak Time Multipliers:
- Morning Rush (7-9am): 2.0√ó wait time
- Lunch Rush (12-2pm): 1.8√ó wait time
- Afternoon Peak (4-5pm): 1.5√ó wait time
- Off-peak: 1.0√ó wait time

Coffee Time Calculation:
Base time (3 min) √ó Peak multiplier = Final time (2-8 min range)

Busy Levels:
- Low (0-30%): 2-3 min, green indicator
- Medium (30-70%): 3-5 min, orange indicator
- High (70-100%): 5-8 min, red indicator

üì° Live Transit Data Processing

Module: data-scraper.js

Data Sources:
- PTV GTFS-Realtime feeds (trains, trams, buses, V/Line)
- Protobuf format parsed to JSON

Update Cycle:
1. TRMNL device requests data every 30 seconds
2. Server checks cache (25-second TTL)
3. If stale: fetch fresh GTFS-RT data from PTV
4. Parse departure times, platforms, delays
5. Cache results and return to device

Data Freshness: Live data max 30 seconds old
Stale Threshold: Shows warning if data > 60 seconds old
